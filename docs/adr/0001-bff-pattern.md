# ADR-0001: BFF（Backend for Frontend）パターンの採用

- **ステータス**: 採用
- **決定日**: 2026-03-15
- **決定者**: アーキテクチャチーム
- **関連ADR**: ADR-0002, ADR-0004

## 背景・課題

フロントエンド（Next.js）が複数のバックエンドマイクロサービスと直接通信する構成を検討した場合、以下の問題が生じる。

- **認証の複雑化**: 各サービスが独自に認証を処理すると、フロントエンドは複数の認証トークンを管理しなければならない
- **CORS問題**: ブラウザからドメインをまたいで複数サービスへ直接アクセスするとCORS設定が複雑化する
- **データ集約の責務**: 1画面の表示に複数サービスのデータが必要な場合、フロントエンド側で並列リクエストと集約処理を実装しなければならない
- **バックエンドサービスとのインターフェース汚染**: バックエンドサービスのレスポンス形式がフロントエンドの都合に引きずられ、サービスの汎用性が失われる

## 決定内容

フロントエンドとバックエンドマイクロサービス群の間に **BFF（Backend for Frontend）レイヤー** を設ける。

```
Browser → Frontend(Next.js) → BFF(NestJS) → Service A
                                           → Service B
                                           → Service C
```

BFFの責務を以下に限定する。

- 認証・認可の一元化（JWT検証）
- 複数サービスへのリクエスト集約と並列実行
- フロントエンド向けレスポンス変換・整形
- 入力バリデーション
- レート制限・セキュリティヘッダー付与

## 理由

- **フロントエンドの簡素化**: フロントエンドは1つのエンドポイント（BFF）とだけ通信すればよい。認証トークン管理も1箇所に集中できる
- **バックエンドサービスの保護**: バックエンドサービスはBFFからのみアクセスを受け付けることができ、インターネットに直接公開しなくて済む
- **レスポンス最適化**: 画面に必要なデータだけをまとめて返す「API集約」が可能になり、ネットワークラウンドトリップを削減できる
- **変更の局所化**: バックエンドサービスのAPI変更をBFFで吸収できるため、フロントエンドへの影響を最小化できる

## 検討した代替案

| 案 | 概要 | 却下理由 |
|---|---|---|
| フロントエンドから直接マイクロサービスを呼ぶ | Next.js API Routes または Client から直接呼び出し | 認証の複雑化・CORS・データ集約の問題が残る |
| API Gateway のみ（BFFなし） | Kong/AWS API GatewayでルーティングのみBFF | フロントエンド向けのレスポンス変換・集約ができない。汎用GWではフロントエンド固有の最適化に限界がある |
| GraphQL BFF | BFFをGraphQLで実装 | チームのGraphQL習熟度・運用コストを考慮し、RESTの方がシンプルと判断。将来的な移行は妨げない |

## 結果・影響

### メリット
- フロントエンドのコードが大幅にシンプルになる（認証・データ集約ロジック不要）
- バックエンドサービスをインターネットから隔離できる
- フロントエンド・バックエンドを独立してデプロイ・スケール可能

### デメリット・トレードオフ
- BFF自体がSPOF（単一障害点）になりうる → 水平スケールで対応
- BFFを経由する分、レイテンシが若干増加する（同一ネットワーク内なら1-5ms程度）
- 管理するサービス数が1つ増える

### 対応が必要なこと
- BFFのヘルスチェックエンドポイント実装（`GET /health`）
- BFFのロードバランシング設定（本番環境）
- BFF → 各サービス間の通信タイムアウト・リトライ設計
