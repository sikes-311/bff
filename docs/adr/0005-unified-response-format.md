# ADR-0005: APIレスポンスフォーマットの統一

- **ステータス**: 採用
- **決定日**: 2026-03-15
- **決定者**: アーキテクチャチーム
- **関連ADR**: ADR-0001, ADR-0002

## 背景・課題

BFFがフロントエンドに返すAPIレスポンスのフォーマットを統一する必要があった。統一しないと以下の問題が生じる。

- エンドポイントごとにレスポンス構造が異なり、フロントエンドの処理が複雑化する
- エラー時のレスポンス構造がバラバラだと、エラーハンドリングのロジックを統一できない
- ダウンストリームサービスのレスポンス形式が変わった際に影響範囲が広くなる

## 決定内容

BFF → フロントエンドのレスポンスを以下のフォーマットに統一する。

**成功（単件）**
```json
{
  "data": { ... },
  "meta": { "timestamp": "2024-01-01T00:00:00.000Z" }
}
```

**成功（一覧）**
```json
{
  "data": {
    "items": [ ... ],
    "total": 100,
    "page": 1,
    "limit": 20
  },
  "meta": { "timestamp": "2024-01-01T00:00:00.000Z" }
}
```

**エラー**
```json
{
  "error": {
    "code": "RESOURCE_NOT_FOUND",
    "message": "指定されたリソースが見つかりません",
    "details": {}
  },
  "meta": { "timestamp": "2024-01-01T00:00:00.000Z" }
}
```

実装方法:
- **成功レスポンス**: `TransformInterceptor`（グローバル登録）が全レスポンスを `{ data, meta }` にラップ
- **エラーレスポンス**: `GlobalExceptionFilter`（グローバル登録）が全例外を `{ error, meta }` に変換

## 理由

- **フロントエンドの一貫性**: `response.data.data` や `response.data.error` という統一したアクセスパターンが使えるため、APIクライアントの実装が単純化される
- **エラーコードの機械判定**: `error.code` を文字列定数（`RESOURCE_NOT_FOUND` 等）にすることで、フロントエンドがHTTPステータスコードだけでなく詳細なエラー種別に基づいて処理を分岐できる
- **`meta.timestamp` の有用性**: タイムゾーン問題のデバッグ、キャッシュ確認、ログとの突合に役立つ
- **AOP（アスペクト指向）的実装**: `TransformInterceptor` と `GlobalExceptionFilter` をグローバルに1回登録するだけで全エンドポイントに適用されるため、各コントローラーで意識する必要がない

## 検討した代替案

| 案 | 概要 | 却下理由 |
|---|---|---|
| JSend規格 | `{ status: "success", data: ... }` のフォーマット | `status` フィールドとHTTPステータスコードの二重管理になる。また一覧のページネーション情報の置き場が曖昧 |
| JSON:API規格 | `{ data: [{ id, type, attributes }], links, meta }` | 規格として完成度は高いが、フロントエンドとBFF間の内部APIに対してはオーバースペック。実装コストも高い |
| 素のレスポンス（フォーマットなし） | コントローラーが直接任意のJSONを返す | エンドポイント間で不統一になり、フロントエンドの型定義が複雑化する |

## 結果・影響

### メリット
- フロントエンドの `ApiResponse<T>` 型が統一され、API呼び出しのコードが簡潔になる
- エラーハンドリングを `response.data.error.code` の一致で統一的に処理できる
- `TransformInterceptor` をOFFにしたい特定のエンドポイント（例: ヘルスチェック）は `@SkipTransform()` デコレータで対応可能

### デメリット・トレードオフ
- 全レスポンスが `{ data: ... }` にネストするため、フロントエンドでは `response.data.data` という二重アクセスになる（axiosの `response.data` + BFF形式の `data`）。API Clientで吸収する
- 既存の外部クライアントが直接BFFを叩く場合はフォーマットへの対応が必要

### 対応が必要なこと
- フロントエンドの `src/lib/api/client.ts` で `response.data.data` を自動的に取り出すインターセプターを設定する（実施済み）
- ヘルスチェックエンドポイント（`GET /health`）はTransformInterceptorの対象外とする設計が必要
