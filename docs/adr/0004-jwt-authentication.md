# ADR-0004: 認証方式にJWT（Bearer Token）を採用

- **ステータス**: 採用
- **決定日**: 2026-03-15
- **決定者**: アーキテクチャチーム
- **関連ADR**: ADR-0001, ADR-0002

## 背景・課題

BFFにおける認証方式を決定する必要があった。フロントエンドからBFFへのリクエストを認証・認可するための仕組みとして、以下の要件がある。

- フロントエンド（ブラウザ）とBFF間の認証ができること
- BFFがステートレスに動作できること（水平スケールに対応）
- アクセストークンの有効期限を短く設定し、リフレッシュトークンで継続的な認証を維持できること
- ダウンストリームサービスへのリクエスト伝播が可能なこと

## 決定内容

認証方式に **JWT（JSON Web Token）Bearer Token** を採用する。

- **アクセストークン**: 有効期限1時間、`Authorization: Bearer <token>` ヘッダーで送信
- **リフレッシュトークン**: 有効期限7日、`POST /auth/refresh` で新しいアクセストークンを取得
- **トークン保存**: `localStorage`（現在の実装）
- **BFF側の検証**: `passport-jwt` によるローカル検証（ダウンストリームへの問い合わせなし）

エンドポイント:
```
POST /auth/login    → accessToken + refreshToken を返す
POST /auth/refresh  → 新しい accessToken を返す
POST /auth/logout   → クライアント側でトークンを破棄（BFFはステートレス）
```

## 理由

- **ステートレス**: JWTはサーバーサイドにセッション情報を保持しないため、BFFを複数台に水平スケールしてもセッション共有が不要
- **自己完結性**: トークン自体にユーザー情報（sub, email, role）を含むため、毎回データベースやユーザーサービスに問い合わせず、BFFローカルで検証できる
- **ダウンストリーム伝播**: BFFから各バックエンドサービスへのリクエスト時に同じJWTをそのまま転送でき、サービス間の認証を統一できる
- **短命なアクセストークン（1h）**: トークン漏洩時のリスクを限定できる。リフレッシュトークン（7日）により UX を損なわない

## 検討した代替案

| 案 | 概要 | 却下理由 |
|---|---|---|
| セッション（Cookie + サーバーサイドセッション） | セッションIDをCookieで管理し、サーバーにセッション情報を保存 | 水平スケール時にセッションストア（Redis等）が必要になり、インフラの複雑性が増す。ただしXSSリスクという観点ではCookieの方が優位 |
| OAuth2 / OIDC（外部認証プロバイダー） | Auth0/Cognito/Keycloak等を使う | 現段階ではオーバースペック。必要になった際の移行を妨げない設計にしてある |
| mTLS（相互TLS） | クライアント証明書による認証 | フロントエンドとBFF間ではオーバースペック。BFF↔バックエンドサービス間の通信では将来的に検討する価値あり |

## 結果・影響

### メリット
- BFFがステートレスになり、インフラがシンプル
- `@nestjs/passport` + `passport-jwt` の組み合わせでBFF側の実装が簡潔
- `JwtAuthGuard` を `@UseGuards()` で付けるだけで認証を強制できる

### デメリット・トレードオフ
- **JWTの失効問題**: アクセストークンを即時無効化できない（有効期限内は有効）。強制ログアウトが必要な場合はトークンブラックリスト（Redis等）の追加が必要
- **`localStorage` のXSSリスク**: `localStorage` はJavaScriptからアクセス可能なため、XSS脆弱性があるとトークンが盗まれる。セキュリティ要件が高い場合は `HttpOnly Cookie` への移行を検討
- リフレッシュトークンの安全な保管はクライアント実装に依存する

### 対応が必要なこと
- **本番環境でのセキュリティ強化（優先度: 高）**:
  - `JWT_SECRET` を十分な長さのランダム文字列にする（32文字以上）
  - `localStorage` から `HttpOnly Cookie` への移行を検討
- **将来対応**:
  - 強制ログアウト要件が発生した場合: Redisによるトークンブラックリスト実装
  - SSO要件が発生した場合: OIDC対応（`passport-openidconnect`）への拡張
